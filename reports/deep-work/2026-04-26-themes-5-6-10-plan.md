## Theme 5 — Extract `_compute_par_klaim` shared PAR helper

### Problem (one sentence)
Klaim PAR is computed in **4 places** with subtly different proxy logic
(`compute_par`, `compute_klaim_covenants` PAR30 + PAR60, `compute_klaim_borrowing_base`
ineligibility), guaranteeing drift over time as one site is fixed without the
others.

### Why it's architectural (not narrow)
Same metric (`PAR30`), same source data, three different code paths producing
three different numbers depending on which API endpoint you hit. The Red Team
report Section A+B Finding #29 calls this out as a class of bugs:

> The `compute_par()` (analysis.py) uses `(active['est_dpd'] >= threshold_days) &
> (active['outstanding'] > 0) & (active['shortfall'] > 0)`. The covenant
> `compute_klaim_covenants` (portfolio.py) uses `(age_active > threshold) &
> (pending > 0)`. The two will diverge under realistic tape conditions —
> same Klaim tape can produce two different PAR30 readings depending on which
> endpoint serves it.

Wave 1+2 didn't tackle this because (a) it's not a single bug, it's a
duplicated-logic problem, (b) the covenant + tape PAR readings happen to agree
*today* on the live data — but every future fix risks divergence.

### Current state (4 sites)
| File | Line | Method | Used by |
|---|---|---|---|
| `core/analysis.py:1708` | `compute_par` (3-method dispatch: shortfall-based est_dpd → Option C empirical → unavailable) | Tape Analytics PAR30+/60+/90+ KPIs |
| `core/portfolio.py:1242-1254` | `compute_klaim_covenants` PAR30 (age > 30 fallback OR age > 90 + pending > 0) | Portfolio Analytics PAR30 covenant |
| `core/portfolio.py:1247-1252` | `compute_klaim_covenants` PAR60 (age > 60 fallback OR age > 120 + pending > 0) | Portfolio Analytics PAR60 covenant |
| `core/portfolio.py:1037-1063` | `compute_klaim_borrowing_base` ineligibility (age > 91d) | BB waterfall ineligible bucket |

Plus parallel duplicate logic for SILQ at `core/analysis_silq.py:122-135` (Tape)
and `:281-295` (covenant) — same drift potential.

### Proposed approach
Single shared helper in `core/analysis.py`:

```python
def _compute_par_klaim(df, mult, ref_date, *,
                      threshold_days,
                      method='auto',  # 'direct' | 'pending_age' | 'shortfall' | 'option_c' | 'auto'
                      lifetime_denominator=False) -> dict:
    """Single source of truth for Klaim PAR. Returns:
        {'amount': float, 'pct': float, 'count': int, 'method': str, 'confidence': str}
    """
```

`method='auto'` chooses the highest-confidence applicable method per Framework
§17. Callers can force a specific method (e.g., covenants need `pending_age` to
match MMA def). Both `compute_par` (Tape) and `compute_klaim_covenants` (Portfolio)
delegate, with the covenant decision keying off the returned `pct`.

Same pattern for SILQ via parallel `_compute_par_silq` helper (Aajil uses
installment DPD which is a different metric, so excluded — Aajil keeps its own).

### Files affected
**Modified (4):**
- `core/analysis.py` — add `_compute_par_klaim` + `_compute_par_silq` helpers; refactor existing `compute_par` to delegate
- `core/analysis_silq.py` — refactor `compute_silq_summary` + `compute_silq_delinquency` to delegate
- `core/portfolio.py` — refactor `compute_klaim_covenants` PAR30 + PAR60 + BB ineligibility branches to delegate
- `tests/test_analysis_klaim.py` + `tests/test_analysis_silq.py` — add helper unit tests

**Net diff:** ~200-300 lines (mostly deletions — duplicate logic collapses).

### Test strategy
1. Pin current behaviour first: snapshot test the existing `compute_par` and
   `compute_klaim_covenants` PAR30 outputs on Apr 15 tape (golden number). The
   refactor MUST preserve them exactly.
2. Add helper unit tests that exercise each method path independently.
3. Add a meta-test that asserts both Tape and Portfolio routes produce identical
   `pct` for the same snapshot (the bug class this fix closes).

### Risk
**Medium.** PAR is a covenant-binding metric; any drift in numerical output
would surface immediately on production dashboards. Mitigation: golden-number
test BEFORE refactor, identical assertions after.

### Sequencing
- **Independent of Themes 6 + 10.** Can ship in any order.
- Recommended: do this BEFORE Theme 5's analogue for Aajil if/when it lands
  (avoid second pattern of divergence).

### Recommended decision point
Greenlight this when next Klaim PAR-related fix lands (compounding interest on
the divergence problem). Estimated effort: 1 focused day.

## Theme 6 — Atomic JSON / JSONL write helper

### Problem (one sentence)
13+ files in `core/mind/` and `core/external/` use `open(path, 'w')` (truncate)
or `open(path, 'a')` (append) without OS-level locking or atomic-rename
semantics — crash mid-write or concurrent write produces corrupt JSON.

### Why it's architectural (not narrow)
Each individual call site is a 2-line patch. But the codebase has 16+ of them,
and three already use the correct atomic pattern (`framework_codification.py`,
`promotion.py`, `relation_index.py`) — so the canonical pattern exists, it's
just not consistently used. Red Team Section E Findings #7, #16, #25 each
flagged a different file from the same pattern.

### Current state
Survey of `core/mind/` + `core/external/`:

| Pattern | Count | Files |
|---|---|---|
| **Atomic (correct)** | 3 | framework_codification.py:198, promotion.py:148, relation_index.py:60 |
| **`open('w')` (truncate, no atomic)** | 9 | analyst.py:79, company_mind.py:489, company_mind.py:582, intelligence.py:274, master_mind.py:742, pattern_detector.py:566, session.py:136, …  |
| **`open('a')` (append, race-prone on Win)** | 4 | asset_class_mind.py:114, company_mind.py:171, compiler.py:138, compiler.py:144, master_mind.py:190 |

Two failure modes:
1. **Crash mid-write** — disk full, OS kill, power loss → JSON file truncated, parse fails on next read
2. **Concurrent writes (Windows)** — `open('a').write` is not guaranteed atomic on Win32. Two pattern detector runs ingesting the same company at once produce interleaved JSONL lines, parse error on next read

### Proposed approach
Extract a single helper module `core/utils/atomic_io.py`:

```python
def atomic_write_json(path: Path, data: dict | list, *, indent: int = 2) -> None:
    """Atomic write: serialize to <path>.tmp, fsync, rename to <path>.
    Cross-platform via os.replace which is atomic on Win + Unix."""

def atomic_append_jsonl(path: Path, record: dict) -> None:
    """Append-with-lock for JSONL files. Uses filelock package for cross-platform
    POSIX-style locking. Single record = single write under the lock."""

def atomic_read_jsonl(path: Path) -> Iterator[dict]:
    """Read JSONL skipping malformed lines (logs WARNING). Tolerates concurrent
    writes that the locks haven't fully prevented (defense in depth)."""
```

Replace all 13 non-atomic call sites with helper calls. Add `filelock` to
requirements.

### Files affected
**Created:**
- `core/utils/__init__.py` (new dir if not present)
- `core/utils/atomic_io.py` (~80 lines)
- `tests/test_atomic_io.py` (~60 lines covering crash sim + concurrent write)

**Modified (~13 mind/external files):**
- All `open('w')` and `open('a')` JSON/JSONL writes → helper calls
- `requirements.txt` → add `filelock>=3.13`

**Net diff:** ~300 lines added, ~50 lines deleted.

### Test strategy
1. Crash simulation: write half a file via direct fd manipulation, kill, verify
   the helper recovers gracefully on next read (skips malformed line)
2. Concurrent writes: spawn N threads writing to the same JSONL, assert all N
   records present and parseable
3. Snapshot existing data files and assert the helper's output matches the old
   format byte-for-byte (no schema change)

### Risk
**Low.** Failure modes are silent; the helper is a strict superset of current
behaviour. Worst case: marginal write latency increase from fsync (negligible
for human-paced JSONL appends).

### Sequencing
- **Independent of Themes 5 + 10.**
- No urgency — current bugs are race conditions that probably haven't caused
  observed production damage. Worth doing before next deploy that bumps
  concurrent-ingest scenarios.

### Recommended decision point
Greenlight after Theme 5 lands. Same effort estimate (~1 day) but lower risk
so doable as a fill-in slot.

## Theme 10 — Extract `run_agent_in_thread` SSE helper + refactor 5 endpoints

### Problem (one sentence)
`backend/main.py:3303` correctly offloads SSE agent runs to a `threading.Thread`
(per Session 32 doctrine), but `backend/agents.py` has **5 SSE endpoints**
(`analyst_stream`, `memo_generate_stream`, `memo_regenerate_section`,
`compliance_check_stream`, `onboarding_analyze`) all using the broken
`asyncio.create_task` pattern that freezes uvicorn's event loop on long sync
Anthropic calls.

### Why it's architectural (not narrow)
Session 32 codified the thread-offload as binding for any SSE agent endpoint:

> backend/agents.py::_stream_agent still uses the broken inline pattern —
> works in the short-call common case but has the same latent bug; refactor
> when touched.

Today the 5 endpoints work in the common path (short Anthropic calls that
return quickly). They WILL freeze when:
- Memo polish runs against Aajil with full tool coverage
- Compliance check on a large facility doc with many extracted covenants
- Memo regen with research packs hitting CF Free's idle window

The fix is one helper extraction + 5 endpoint refactors. Without it, each new
SSE endpoint copies the broken pattern.

### Current state
`backend/agents.py` — 5 SSE endpoints, all share `_stream_agent` (line 116)
which uses `asyncio.create_task(_producer())`:
- `analyst_stream` (line 191) → `_stream_agent` → asyncio.create_task
- `memo_generate_stream` (line 257) → custom impl, also asyncio.create_task at line 145
- `memo_regenerate_section` (line 387) → `_stream_agent`
- `compliance_check_stream` (line 419) → `_stream_agent`
- `onboarding_analyze` (line 482) → `_stream_agent`

`backend/main.py:3244-3303` — 1 SSE endpoint (`get_executive_summary_stream`)
that does the right thing: dedicated `threading.Thread` with own asyncio loop
and `loop.call_soon_threadsafe` on the queue.

### Proposed approach
Extract the thread-offload into a reusable helper:

```python
# core/agents/sse_helpers.py

async def run_agent_in_thread(
    runner: AgentRunner,
    message: str,
    session: SessionState,
    *,
    on_event: Callable[[StreamEvent], Awaitable[None]] | None = None,
    heartbeat_interval_s: float = 20.0,
) -> AsyncIterator[StreamEvent]:
    """Run an agent's stream in a dedicated thread, yielding StreamEvents on
    the main event loop via asyncio.Queue.

    Handles:
    - Thread-local asyncio loop creation
    - loop.call_soon_threadsafe queue puts
    - 20s heartbeat emission for CF Free idle-cap survival
    - Clean shutdown on consumer disconnect (cancel + join)
    - Error propagation through queue (StreamEvent("error", ...))

    See Session 32 lessons.md for the binding architectural rationale.
    """
```

Refactor the 5 SSE endpoints to use it. CI test: assert no SSE endpoint in
`backend/agents.py` references `asyncio.create_task` (regression guard).

### Files affected
**Created:**
- `core/agents/sse_helpers.py` (~120 lines incl heartbeat + cleanup)
- `tests/test_sse_helpers.py` (~80 lines — thread offload contract, heartbeat
  cadence, error propagation, queue full backpressure)

**Modified:**
- `backend/agents.py` — refactor `_stream_agent` (line 116) and 5 endpoints to
  delegate to helper. Net deletion ~80 lines.
- `backend/main.py:3244-3303` — refactor `get_executive_summary_stream` to use
  helper (already correct pattern; this is just dedup).
- `tests/test_agent_rate_limit.py` + `tests/test_agent_runtime.py` — verify
  rate-limit/circuit-breaker behaviour preserved.

**Net diff:** ~200 lines added, ~150 lines deleted.

### Test strategy
1. **Thread offload contract test**: synthetic agent that blocks 5s in a sync
   call. Without offload, asyncio loop freezes (heartbeat doesn't fire);
   with offload, heartbeat fires every 20s.
2. **Disconnect cleanup**: simulate client abort mid-stream, assert worker
   thread joins within 1s and no leaked StreamEvent in queue.
3. **CI regression guard**: lint test that `grep "asyncio.create_task" backend/agents.py`
   returns nothing.
4. **Backward compat**: existing test_agent_rate_limit + test_agent_runtime
   must pass unchanged.

### Risk
**Medium-high.** SSE plumbing touches the production browser UX directly. A
regression here = blank streaming output on the live UI. Mitigation:
1. Land helper + tests in isolation first (no endpoint changes)
2. Refactor one endpoint (`memo_regenerate_section` — least exercised) and
   manually browser-verify before continuing
3. Refactor remaining 4 endpoints in a single follow-up commit

### Sequencing
- **Independent of Themes 5 + 6.**
- Highest urgency of the 3 themes — has known production failure modes
  (Aajil exec summary freeze documented in CLAUDE.md "Key Architectural
  Decisions" Session 32 entry).

### Recommended decision point
Greenlight FIRST among the three themes. ~1.5 days for helper + 5 endpoint
refactors + browser verification.
